## 1. ★GLM併用体制(今後ずっと使う・最重要インフラ)

### 1.1 何を作ったか
- **GLM Coding Lite 契約**($16.2/月。z.ai)。GLM-5.2を**Claude Code CLIに乗せ換え**て実装役にする。
- **`glm` コマンド**: 本物Claudeは`claude`のまま、`glm`で起動すると中身だけGLM-5.2になる。`~/.zshrc`に関数、APIキーは`~/.glm_key`(git管理外・600)。
- **agmsg**(エージェント間メッセージ・共有SQLite): チーム`pchamdb` に `claude-design`(本物Claude=設計/検証) / `glm-impl`(GLM=実装) / `gemini-impl`(無料Gemini=予備)。
- 起動メモ = `~/glm_起動メモ.txt`(`cat ~/glm_起動メモ.txt`で確認)。

### 1.2 ★起動・運用のやり方(次回もこれ)
```bash
# ターミナル①= 設計役(本物Claude)
cd "/Users/masamichi/Documents/ポケモンDB" && claude
# ターミナル②= 実装役(GLM-5.2)
cd "/Users/masamichi/Documents/ポケモンDB" && glm
```
- **再設定不要**: glm関数/APIキー/agmsgチーム登録は全部保存済。再joinも不要。
- **2台を会話させる**: claude側に「glm-implに『○○して』と送って」/ glm側に「agmsgのinboxを確認して」。
- GLMはキー保存済で何もいらない。Geminiだけ使う時に`export GEMINI_API_KEY=...`が要る(普段無視でOK)。
- **GLM使用量**: z.ai → 左メニュー「GLM Coding Plan」→ Usage。
- ★**GLMは「auto mode on」だと許可なしで自律実行**。仕様を渡せば自分でファイル作成まで完遂する。進捗の逐次relayは不要。

### 1.3 ★GLMに作業させる正攻法(今回の実例=ability_all.html)
1. 私(claude-design)が**仕様をagmsgで送る**(`send.sh pchamdb claude-design glm-impl "..."`)。データの場所・要件・手順・完了報告先を明記。
2. ユーザーがGLM画面で「agmsgのinbox確認して」→ GLMが**自律でクローン元を読み→実装→検証**。
3. GLMが「できた」と返信 or 私がループで検知 → **私がPDCA検証**(実機・件数・JSエラー0・i18n)。
4. OKなら私からGLMに「検証OK」返信。ダメなら差し戻し。
- **役割分担**: GLM=作る(実装・量産・機械作業) / Claude=設計・声判定・検証 / 阿部さん=声の最終OK。**声/taste判断はGLMに丸投げしない**(機械ゲートで弾ける作業だけGLM)。

### 1.4 ★ループで自動監視(relayを無くす)
GLMが長い実装をしている間、**`ScheduleWakeup`で自分を数分おきに起こして**「ファイルできた? inbox連絡来た?」を自動チェック→完成したら自動検証→未完なら再スケジュール。
- delaySecondsは270秒(キャッシュ窓内)が基本。プロンプトに「ファイル存在確認→inbox確認→完成なら検証/未完なら再ScheduleWakeup/N回超で『GLM止まってるかも』とユーザー確認」を書く。
- 今回これで**ユーザーのrelayゼロでability_all.html完成を自動検知**できた。

### 1.5 ★agmsg monitorモード = ON にした(2026-07-01末)
- このプロジェクトを **monitorモードに設定済み**(`delivery.sh set monitor claude-code "$(pwd)"`。claude-code型=claude/glm両方に効く)。`.claude/settings.local.json`にSessionStartフック登録済。
- **効果**: 次回 claude / glm を起動し直すと watcher が自動起動し、相手のメッセージが**リアルタイムで自動受信**される(「inbox確認して」もループも不要)。GLM⇄Claudeが本当に会話として流れる。
- ★**今動いてるセッションには未適用=次回起動で有効化**。
- 確認: `delivery.sh status claude-code "$(pwd)"`(mode: monitor) / OFFに戻す: `set off` か `set turn`。
- → 次回からは「ループ自動監視」ではなく**monitorのリアルタイム自動連携**が基本。ループはmonitorが使えない場面のフォールバック。

## 4. ★ワークフロー/ループの回し方(再利用知見)
- **全数照合パターン**: 427技を15技/組×29グループのファイルに書き出し→各エージェントが**自分のグループファイルだけRead**して照合(argsで全データを渡すとトークン爆食い→ファイル分割が効率的)。web不要の「新訳↔ヤック意味照合」なら安い。
- **トークン節約の鉄則**: いきなり全数web検証(~1400万tok)せず、**①無料ヒューリスティック/機械チェックで"あやしい技"を絞る→②少数だけweb→③客観確認は小サンプル**。今回ヒューリスティックで222→39に絞れた。
- ★ただし**「サンプルOK」は偽の安心**: 10/10 correctでも全数照合で99件出た。**系統バグは全数でないと出ない**。最終品質確認は全数で。
- **Workflowの罠**(再掲): ①`require`不可 ②`args`は文字列で来る→`typeof args==='string'?JSON.parse(args):args`ガード ③`Date.now/Math.random`不可 ④model:'sonnet'で量産(コスト)。
- **ループ(ScheduleWakeup)**: GLMやバックグラウンド処理の完了を自動監視。270秒間隔・自己再スケジュール・N回超で人間に確認。

## 5. ★注意すること・教訓・間違ったこと(必読)
1. **★出発点を絶対に逆にしない**(不変): effects(sim土台)→compose(訳す)。effects空で説明文だけ手書きは禁止。語彙外メカは新kindを足す。([[effects-sim-phase-first]])
2. **★全数照合 > サンプル/ヒューリスティック**: 今回の最大の学び。「ほぼ完成」と思っても**全数で系統バグが大量に出る**。ユーザーの「全部ちゃんと?」が正しかった。最終確認は必ず全数。
3. **web判定のノイズ**: エージェントは「第9世代SVで使用不可」を多数flagするが、**全国版=全部入りなので無関係**(無視)。プロンプトで明示的に除外指示する。
4. **ヤック自体にバグがある**: foresight/miracle-eyeでヤックがタイプ名を落としていた等。**web真実 ↔ ヤック ↔ 訳の三者照合**が最強。ヤックだけをゴールデンにしない([[バトル再現_羅針盤]]の自己参照回避と同じ)。
5. **national技はeffects[]格納**: 旧タグ/旧ロジックが bd.crit_stage 等のフラット項目を読む箇所は、national技で漏れる。effectsからの補完が必要(タグ・優先度・範囲など)。
6. **キャッシュバスター必須**: JS(waza_picker.js等)を変更したらHTMLの`?v=`をbumpしないとブラウザが古いJSを読む。
7. **GLMはタスクをagmsgのinboxに置くだけでは動かない**: ユーザーがGLM画面で「inbox確認して」を打つ必要(monitorモード未設定時)。だから**ループ自動チェック**が有効。
8. **GLMのピーク制限**: 「Currently in peak hours, switch to GLM-5-Turbo」が出ることがある。9月までオフピーク1xプロモ。重い実装は時間帯注意。
9. **声/tasteはGLMに丸投げしない**: 機械ゲートで弾ける作業(量産・構造・タグ)だけGLM。説明文の声・ヤックン耳テストは私＋阿部さん。
10. **全部ローカル・未push**: 本番反映は阿部さんOK後。声の最終判定は阿部さんの耳=Claudeは★→✓に上げない。共有docのcommitは検証担当の了解。

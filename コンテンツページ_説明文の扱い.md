# コンテンツページ:技の説明文の扱い(決定メモ)

決定日: 2026-06-12 / 阿部さん

## 決定(暫定運用)

- **静的コンテンツページ(/pokemon/・/move/)の技一覧には、説明文を載せない。** 事実データ(タイプ・分類・威力・命中・PP)だけを出す。
- 説明文は**暫定的に「無し」でよい**。今は出さないことを正とする。
- 技の個別ページ(/move/<key>.html)は当面**作らない**(`MOVE_PAGES = false`)。

## なぜ(出してはいけない理由)

ページ生成で一度使っていた `WAZA_MAP[].description`(古い短文フィールド)は、出すと危険:
1. **11技が `description_legacy`(ヤックン)と完全一致** → 丸写し=盗用リスク。北極星「独自に作る」に反する。
2. 声が機械寄り(「HP/4」「威力1/2」「音技」等)で、子ども口調になっていない。
3. **SSOTでない。** sim と連動する本来の真実(`battle_data.effects`)とは別物の、古い独立フィールド。

## 正しいSSOT(将来こう繋ぐ)

```
battle_data.effects(唯一の真実)
   ├─→ バトルシミュレーター(sim が読む)
   └─→ compose エンジン(tools/_waza_compose.js の compose()) → 説明文
```
- `compose()` は effects から説明文を生成する**唯一のエンジン**(sim と同じ意味を日本語化)。
- 現状は育成中(例: なみのり = `{text:'', holes:['半無敵命中']}` = まだ喋れない)。
- **段取り**: sim作業で effects が育つ → compose が技を喋れる → 技リストに反映 → **阿部さんの耳で✓** → そのとき初めて、ページ生成器が **compose出力(=一箇所)** を読むよう繋ぐ。`MOVE_PAGES=true` で個別ページも開通。
- これで「一箇所変えたら全部変わる」になる(データの二重持ちをしない)。

## 実装メモ
- 生成器 `tools/_gen_content_pages.js`: `movesTable()` と `genMoveIndex()` から説明列を削除済み(2026-06-12)。`description` フィールドは**参照しない**。
- 関連設計: `コンテンツページ_URL設計.md`

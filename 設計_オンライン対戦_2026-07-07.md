# リアルバトル オンライン対戦 設計メモ(調査・設計フェーズ)

作成: 2026-07-07 / 状態: **調査・設計のみ(未実装)** / 依頼: 阿部さん「まずは調査・設計だけ」

---

## 0. 結論(先に)

**実装できる。** ポケモンのバトルは**ターン制**なので、リアルタイム同期がシビアなアクションゲームより格段に作りやすい。今の作り(GitHub Pages 静的サイト)はそのまま残し、**リアルタイム通信の部分だけ外部サービスを足す**。

さらに、うちのエンジンは**オンライン対戦に有利な構造**をしている:
- 行動入力は `selectedMoveIdx` / `switchChoice` / `megaChoice` を各サイドに立てて `runTurn()` を呼ぶだけ(3つの値を送り合えばよい)。
- 結果は全部 `battleLog`(420箇所のlog)に積まれ、クライアントは既存の `say()`/`lineWithFx`/`translateLogLine` で**そのまま再生**できる。
- 乱数(`Math.random`)を**39箇所**使う=クライアント間で独立に計算すると結果がズレる → **「部屋主(ホスト)が計算して、結果のbattleLogを相手に配る」ホスト権限方式が最適**。相手側はエンジンを回さず、送られてきたbattleLogを再生するだけ。**既存の演出・多言語がそのまま効く。**

## 1. アーキテクチャ(推奨=ホスト権限リレー方式)

```
[プレイヤーA(部屋主/ホスト)]        [リアルタイム仲介]         [プレイヤーB(ゲスト)]
 GitHub Pagesのreal_battle.html  <-->  WebSocket/Realtime  <-->  GitHub Pagesのreal_battle.html
 ・エンジン(__sim)を回す=権威         (合言葉ルーム/presence)     ・エンジンは回さない
 ・両者の行動を集めてrunTurn()                                    ・送られたbattleLog差分を
 ・battleLog差分を相手に配信                                         say()で再生(演出/多言語込み)
```

- **GitHub Pagesは引っ越し不要**。HTML/JS/画像は今のまま。追加するのは「リアルタイムの通り道」1つだけ。
- **ホスト権限方式の利点**:
  1. 乱数の同期問題が消える(権威は1つ=ホストのエンジン)。
  2. ゲスト側はエンジン改変ほぼ不要=**送られたログを再生するだけ**。既存の `say()` / `lineWithFx` / `translateLogLine` / 各fx(みがわり・ばけのかわ・タスキ等)が全部そのまま動く。
  3. 実装の大半が「行動3値の送受信」と「battleLog差分の送受信」に集約=小さい。
- **弱点**: ホストがエンジンを持つので、理論上ホストは不正できる(結果を改ざん)。**友達同士・身内対戦なら十分**。公開マッチで公平性を担保するなら「サーバー権限方式」に格上げ(§7)。

## 2. 合言葉(ルームコード)対戦の流れ ← 最初に作るべきもの

1. Aが「オンライン対戦」→「部屋を作る」。チームを組む。**合言葉(例: `pikapika`)or 自動生成コード**が出る。ホスト=A。
2. Aは待機(「相手を待っています…」)。
3. BがPC/スマホの別ブラウザで同ページ→「部屋に入る」→合言葉入力→接続。
4. 接続確立で**チーム情報を交換**(A→B, B→A に slotVal/effort/nature/item/ability/moves を送る)。両者が相手チームをセット。
5. 各ターン:
   - 両者が行動(技/交代/メガ)を選ぶ→**行動3値(selectedMoveIdx or switchChoice, megaChoice)を仲介経由でホストへ**。
   - ホストが両者の行動を `S.sides.self/opp` にセット→`runTurn()`→新規`battleLog`差分を取得。
   - ホストが**battleLog差分をゲストへ配信**。
   - 両者とも `say(差分.map(lineWithFx))` で再生(演出・音・多言語そのまま)。
6. 死に出し・メガ・ゲームオーバーも同じ経路(battleLog差分で表現済み)。

**このモデルなら「阿部さんがPCで部屋を作り、スマホの別ブラウザから合言葉で入ってテスト」がそのまま実現できる。**

## 3. メッセージ・プロトコル(最小)

ルーム内でやり取りするJSONイベント(仲介サービスのメッセージに乗せる):

| type | 送信者 | 中身 | 用途 |
|---|---|---|---|
| `hello` | 両者 | `{name, ready}` | 入室・presence |
| `team` | 両者 | `{size, val, item, ability, gender, nature, effort, moves, lead}` | チーム交換(saveTeamの形と同じ) |
| `action` | ゲスト→ホスト | `{turn, kind:'move'|'switch'|'mega+move', moveIdx?, switchIdx?, mega?}` | その turn の行動 |
| `turnResult` | ホスト→ゲスト | `{turn, log:[{code,msg}...]}` | runTurn後のbattleLog差分 |
| `faintReplace` | 該当者→ホスト | `{benchIdx}` | 死に出しの選択 |
| `ping`/`pong` | 両者 | `{t}` | 生存確認・切断検知 |
| `resign`/`bye` | 両者 | — | 降参・退出 |

- ホストは自分の行動はローカルで、ゲストの`action`を待って揃ったら`runTurn()`。
- **タイムアウト**: 一定時間(例: 60秒)行動が来なければ「時間切れ=ランダム行動 or 降参」を選べる規約に。

## 4. リアルタイム仲介サービスの比較

| サービス | 方式 | 自前サーバー | 無料枠 | 相性 | 備考 |
|---|---|---|---|---|---|
| **PartyKit** | WebSocket(部屋=Durable Object) | 不要(デプロイのみ) | あり(小規模無料) | ◎ | 「部屋」概念が主役=合言葉対戦に最適。ホスト権限もサーバー権限も書きやすい。Cloudflare上。 |
| **Supabase Realtime** | WebSocket(Broadcast/Presence) | 不要(SaaS) | 寛大 | ◎ | presence(待機一覧)が標準装備=ロビーに強い。認証/DBも付いてくる。 |
| **Firebase Realtime DB / Firestore** | ロングポーリング/WS | 不要(SaaS) | 寛大 | ○ | 枯れてて簡単。部屋=ドキュメントに両者が書き込む。presenceは自作寄り。 |
| **Cloudflare Durable Objects** | WebSocket | 要デプロイ(Workers) | あり | ◎ | PartyKitの下地。自前で組むなら最も自由・安い。学習コスト中。 |
| **Ably / Pusher** | WebSocket(pub/sub) | 不要(SaaS) | 小〜中 | ○ | 手軽だが無料枠が小さめ。presenceあり。 |
| **WebRTC(P2P)** | ブラウザ直結 | signalingは要 | — | △ | 中継負荷ゼロだがNAT越え(STUN/TURN)と実装が重い。ターン制には過剰。 |

### 推奨
- **①合言葉対戦(最初の一歩)**: **PartyKit** か **Supabase Realtime**。
  - 「部屋=ルームコード」を素直に書けるのは **PartyKit**。
  - 「待機一覧(presence)」まで見据えるなら **Supabase**(ロビーが楽)。
- どちらも**自前サーバー24時間稼働は不要**、無料枠で個人利用は十分。

## 5. 乱数・確定性の扱い(重要)

- エンジンは `Math.random` を39箇所で使用(急所・命中・追加効果・AI等)。
- **ホスト権限方式なら乱数同期は不要**=ホストのエンジンだけが結果を作り、ゲストは結果ログを再生するだけ。**ここが今回いちばんの設計上の勝ち筋。**
- 将来サーバー権限方式にする場合も同じ(サーバーのエンジンが権威)。

---

## 6. 必要なエンジン/クライアント改修(小さい)

- **エンジン(real_battle_simulator.html)**: ほぼ無改変でよい。`runTurn()`・行動3値・`battleLog`をそのまま利用。
- **real_battle.html(クライアント)**:
  - オンライン用の状態管理(ホスト/ゲスト・接続・部屋)を足す。
  - 通常の `doTurn()` は「AIの行動→runTurn→say」だが、オンラインでは
    - ホスト: 「自分の行動 + ゲストのaction待ち → runTurn → battleLog差分をsay + 配信」。
    - ゲスト: 「自分のactionを送信 → turnResult受信 → say(差分)」(runTurnは呼ばない)。
  - チーム交換(既存の saveTeam/loadTeam のデータ形をそのまま送受信)。
  - 切断・タイムアウト・降参のUI。
  - 演出・多言語・スプライトは**全部そのまま流用**(battleLog再生なので)。

---

## 7. 不正対策とアップグレード経路

- **フェーズ①(身内・友達)= ホスト権限**: ホストが結果を作る。改ざんは理論上可能だが身内なら問題なし。**まずここで動くものを作るのが最短。**
- **フェーズ②(公開・公平)= サーバー権限**: エンジンをサーバー側(Node/Workers)で動かし、両者の行動を受けてサーバーが `runTurn()`→結果を両者へ。クライアントは表示専用。
  - うちのエンジンはブラウザ用のグローバル依存(DOM等)を持つ箇所があるため、**エンジンのコア(計算部分)をDOM非依存に切り出す**作業が必要=ここが公開対戦の一番の山。
  - PartyKit/Durable Objects/Workers なら同じJSをサーバーでも走らせやすい。

## 10. リスク・注意

- **エンジンのDOM依存**: P4(サーバー権限)で問題化。P1〜P3(ホスト権限)は影響なし。
- **切断/離脱**: ターン制なので致命的でないが、タイムアウト規約(60秒等)と再接続 or 判定勝ちのルールが要る。
- **チート**: P1はホスト信頼前提。公開はP4必須。
- **同時接続数/スパム**: 公開時はレート制限・簡易モデレーションが必要。

## ★北極星(最優先・説明文に関わる全作業に適用)

全説明文を「**子どもが、うちの出力だけ読んで、ヤックン(=既存の良い説明文 `description_legacy`)と同じ"意味"に戻れる**」状態にする。**ただし文言はヤックンのコピーではなく独自に作る**(このプロジェクトの出発点＝既存サイトの丸写しでなく独自に作る)。legacy は**意味の参照**であって**文言の正解ではない**。**verbatim一致は盗用リスク＝避ける**。技の説明文は独自作成／特性(ABILITY_DESC)は公式準拠なのでそのまま可。
**データの綺麗さ・構造の完成度・英語ゼロ・カバー率は、手段であって目的ではない。**

**唯一の合否 = ヤックン耳テスト**: 何か作ったら必ず、作る姿勢を捨てて読者の姿勢に座り直し、予備知識ゼロの子のつもりで「うちの出力だけ」を読み、`description_legacy` と並べて ①**意味が戻るか**(legacyと同じ内容に戻れるか) ②**声がやさしい子ども口調か**(漢字を勝手にひらがなに開かない・過剰softening禁止) ③**機械が漏れてないか**(`true`/`0.125`/英語/キー名) ④**legacyの文言コピーになってないか**(verbatim一致=盗用リスク=NG。独自の言葉で・用語は開いて説明=独自化のレバー)。一つでもズレたら未完。grepゼロ・カバー率・"全部日本語になった"・**"legacyとそっくり"は偽の頂上**(そっくり=むしろ赤信号)。
> **直訳テスト(独自性の確認)**: legacy を**見ずに** effects だけから「初見の他人ならどう訳すか」を生成し、それが①意味に戻れるか確かめる。legacyに寄せて作るのでなく、データから独立に作る。

**必ず陥る罠**: 「機械(構造・データ)を完成させる実感」が「ヤックンに戻れるか確かめる規律」に勝つ。**作る最中のセッションは自分の漂流を感じない。** 作り直す前に必ず「**もう `description_legacy` に在るのでは?**」を問え(既存を壊すな)。

**境界線(判断するな・止まってユーザーに聞け)**: 声の最終判定はユーザーの耳。SSOT(legacy/description/effects)が食い違って決着しない・好みでしか決まらない・事実が要る所では、記憶で断言せず権威ソース確認 or ユーザーに聞く。スキーマを推測で足すな。

→ 具体例の山・境界線の詳細は **`ヤックン耳_判断ログ.md`** を**作業前に必ず読む**(原則を解釈せず、具体例にパターンマッチしろ)。

## ★わざリスト説明文の基準(セッションをまたいでも不変・必読)

**技(わざ)の説明文を作る・直す・チェックする時は、必ず `review/rules.html`(元データ=`tools/_rules.js`)の全ルールを基準にする。** これは引き継ぎ後も**変えない確定基準**。
- **★作業手順(セッション引き継ぎ)= `わざ説明文_開通手順.md`**: 「続きをやって」と言われたら**まずこの手順書のとおり**に1ブロック(効果kind)ずつ開通する。STEP1〜10・注意事項・コマンド早見表あり。迷ったらここに戻る。
- **ルールの集約=唯一の基準**: `review/rules.html`(書き方のルール＋言葉のルール)。二重管理しない(ルール本体はここ／`tools/_rules.js`)。
- **確認の場**: `review/waza_list_confirm.html`(`tools/_waza_list_confirm.js`)。本番デザイン・**効果カテゴリー別セクション**で、各技の「**効果(新版・エンジン生成)↔ヤック(legacy)**」を**意味で照合**する。
- **エンジン**: `tools/_waza_compose.js`(effects→説明文)。effects は **simと説明文の二役**。**voiced≠complete**(カバー率はデータ欠けを見ない=意味照合は人間/検証)。
- **略さない(意味が要る所)**: 「短くて済む」より「意味が戻る」。例(必中・2026-06-07 阿部さん確定): ✗「必ず命中する」だけ → ✓「**相手の回避率や自分の命中率に関係なく**、必ず命中する」(かいひ上昇/命中率低下を無視して当たる肝・simにも効く→effectsにも `ignores` を入れる)。
- **照合して抜けを丁寧に追加(各技でやる)**: 新版とヤックンを**見比べ**、新版に抜けている意味があれば、**丁寧に effects へ追加**して補完する(=略さないの実行・voiced≠completeの実践)。例(なみだめ): legacyの「守り技を貫通(ダイウォール除く)」が effects に無→`まもり貫通`追加。
- **能力ランク=±N表記**: 能力ランクの増減は「こうげき-1、とくこう-1 / こうげき+2」で固定(✗「1段階さがる」)。急所ランクは別(「ひとつ上がる」)。
- **役割モデル(運用)**: ①ビルド=作る ②検証=独立に試訳/データ裏取りして**事実をco-sign**(別Claudeの意見に錨を打たない) ③**声の最終判定は人間(阿部さん)の耳**。**Claudeは★→✓に上げない**(耳は人間のみ)。**台帳/ハンドオフ等 共有docのcommitは検証担当の了解**を取る(阿部さん直接指示時を除く)。
- 判断ログ(具体例)・境界線は `ヤックン耳_判断ログ.md`。

## ポケモンデータの参照ルール

**マスターDB (SSOT)**: `pokechan_data.js`

ポケモン本体は `pokechan_data.js` の `const POKEMON_LIST`(同ファイル内で `const DATA = POKEMON_LIST` のエイリアスあり)。技は `WAZA_MAP`、性格は `NATURES`、タイプは `TYPES`/`TYPE_COLORS`。`pokemon_db_v9.html` など各ページはこの定数を読み込んで参照しているだけで、データ本体は持たない。

タイプ・種族値・特性・技学習などは **このDBが正**。記憶ベースの「公式情報」を優先しない。  
不一致を見つけたら、まず **マスターDB を信頼** して、自分の記憶が間違っている前提で対応する。

## macOS 障害対策

ファイルが開けない症状 (Numbers/Excel ハング系) が出たら:

```bash
killall cfprefsd
PID=$(pgrep -x ScopedBookmarkAgent) && kill -9 "$PID"
```

それでもダメな場合: `_review/html_tables/index.html` を Chrome で開く。

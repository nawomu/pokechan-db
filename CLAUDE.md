## ★北極星(最優先・説明文に関わる全作業に適用)

全説明文を「**子どもが、うちの出力だけ読んで、ヤックン(=既存の良い説明文 `description_legacy`)の意味と声にそのまま戻れる**」状態にする。
**データの綺麗さ・構造の完成度・英語ゼロ・カバー率は、手段であって目的ではない。**

**唯一の合否 = ヤックン耳テスト**: 何か作ったら必ず、作る姿勢を捨てて読者の姿勢に座り直し、予備知識ゼロの子のつもりで「うちの出力だけ」を読み、`description_legacy` と並べて ①意味が戻るか ②声が同じか(`『』`で用語を囲む・漢字を勝手にひらがなに開かない・`8分の1`化や過剰softening禁止・ヤックンの表記=`1/8`等を保つ) ③機械が漏れてないか(`true`/`0.125`/英語/キー名)。一つでもズレたら未完。grepゼロ・カバー率・"全部日本語になった"は**偽の頂上**。

**必ず陥る罠**: 「機械(構造・データ)を完成させる実感」が「ヤックンに戻れるか確かめる規律」に勝つ。**作る最中のセッションは自分の漂流を感じない。** 作り直す前に必ず「**もう `description_legacy` に在るのでは?**」を問え(既存を壊すな)。

**境界線(判断するな・止まってユーザーに聞け)**: 声の最終判定はユーザーの耳。SSOT(legacy/description/effects)が食い違って決着しない・好みでしか決まらない・事実が要る所では、記憶で断言せず権威ソース確認 or ユーザーに聞く。スキーマを推測で足すな。

→ 具体例の山・境界線の詳細は **`ヤックン耳_判断ログ.md`** を**作業前に必ず読む**(原則を解釈せず、具体例にパターンマッチしろ)。

## ポケモンデータの参照ルール

**マスターDB (SSOT)**: `pokechan_data.js`

ポケモン本体は `pokechan_data.js` の `const POKEMON_LIST`(同ファイル内で `const DATA = POKEMON_LIST` のエイリアスあり)。技は `WAZA_MAP`、性格は `NATURES`、タイプは `TYPES`/`TYPE_COLORS`。`pokemon_db_v9.html` など各ページはこの定数を読み込んで参照しているだけで、データ本体は持たない。

タイプ・種族値・特性・技学習などは **このDBが正**。記憶ベースの「公式情報」を優先しない。  
不一致を見つけたら、まず **マスターDB を信頼** して、自分の記憶が間違っている前提で対応する。

## macOS 障害対策

ファイルが開けない症状 (Numbers/Excel ハング系) が出たら:

```bash
killall cfprefsd
PID=$(pgrep -x ScopedBookmarkAgent) && kill -9 "$PID"
```

それでもダメな場合: `_review/html_tables/index.html` を Chrome で開く。

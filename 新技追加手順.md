## 🎯 北極星(絶対のルール)

**`description` は legacy(やっくん)のコピーでは絶対NG。**
**effects(SSOT)から compose関数が独自表現で自動生成するのが正解。**

### CLAUDE.md より:
> 全説明文を「子どもが、うちの出力だけ読んで、ヤックンと同じ"意味"に戻れる」状態にする。
> **ただし文言はヤックンのコピーではなく独自に作る**(verbatim一致=盗用リスク=NG)。
> legacy は**意味の参照**であって**文言の正解ではない**。

### 三層の役割
| 層 | 何を書くか | 誰が書くか |
|---|---|---|
| **effects(SSOT)** | バトルシミュレーターが読む構造化データ。フェーズ・条件・確率・倍率 | 人間(調査して構造化) |
| **compose case** | effects の kind/condition/multiplier を日本語に訳すロジック | 人間(_waza_compose.js に case 追加) |
| **description** | compose の自動生成出力。これが最終の説明文 | **compose関数(自動)** |
| description_legacy | やっくん風の参考(意味照合用・コピー禁止) | データから手動 |

### 「effects はバトルシミュレーター前提=常にフェーズを意識して作る」
> effects は**バトルシミュレーター(`real_battle_simulator.html`・Phase構造v7)が読み取る前提で作られたデータ**。
> 説明文(compose)は「**simが読む effects をただ日本語に訳しているだけ**」。
> compose が"賢く"織り込み・並べ替えをしたくなったら、それは effects の phase 設計が足りないサイン。

## 📋 新技追加の正しい手順(必ずこの順で)

### Step 1: 外部調査(専用技/シグネチャー技)
1. 各新ポケモンの専用技を WebSearch / WebFetch で調査(GameWith / yakkun / Bulbapedia / game8 等から最低2サイト)
2. WAZA_MAP に**既存かどうか**を確認(`Object.values(W).map(m => m.name)` と照合)
3. 既存にあれば learners 追加のみ。なければ新技として追加へ進む
4. 並列ワークフロー化推奨(`tools/_research_mb_signature_moves.js` のような形式)

### Step 2: effects の構造化(SSOT)
sim が読む `battle_data` を**正確に**書く。**説明文を先に書いてはいけない**。

```js
battle_data: {
  priority: 1,  // 優先度(先制技なら +N)
  crit_stage: 0, must_crit: false, crit_changes: [],
  effects: [
    // 各効果を kind 別に
    { kind: '能力ランク変化', target: 'opponent', stat: 'special_attack', stages: -1, phase: 'on_use', prob: 100 },
    { kind: 'ひるみ', target: 'opponent', phase: 'on_use', prob: 50 },
  ],
  fails_if: [{ type: 'no_damage_dealt_last_turn' }]  // 失敗条件は ここに(effects ではない)
}
```

#### よく使う kind
- `状態付与`(まひ/ねむり/やけど/どく/もうどく/こおり/こんらん/メロメロ/バインド)
- `ひるみ`
- `能力ランク変化`(stat/stats・stages・prob・phase)
- `威力可変`(basis/formula/per_stage)
- `条件威力倍率`(condition・multiplier)
- `ランク反転`(2026-06-19新規・カラマネロひっくりかえす)
- `ランクリセット` / `ランクコピー`
- `自分拘束`(交代不可)
- `みがわり貫通` / `まもり貫通`

#### 失敗条件(`bd.fails_if[]`に書く・effectsではない)
- `current_hp_at_or_below_fraction`(みがわり等)
- `no_damage_dealt_last_turn`(どげざつき・2026-06-19新規)
- `target_not_selecting_attacking_move`(ふいうち)
- `user_not_type`(もえつきる)

### Step 3: compose case の対応確認・追加
`tools/_waza_compose.js` を開いて、effects の kind がすでに case にあるか確認:

```bash
grep -n "case '<kind名>'" tools/_waza_compose.js
```

#### 対応していない場合
新しい case を追加。**legacy をコピーしない・独自表現で・simの動作を訳す**。

例(2026-06-19 追加した3つ):
```js
case 'ランク反転':
  return `相手の能力ランクの上がりさがりを、すべて反対にする(+1なら-1、-2なら+2)`;

case '威力可変': {
  if (e.basis === 'rage_accumulator') {
    const per = e.per_hit_taken || 50;
    const reset = e.reset_at_turn_end ? '。ターンが終わると蓄積はリセットされる' : '';
    return `攻撃を受けるたびに威力が${per}ずつ上がっていく${reset}`;
  }
  // ... 他のbasis
}

// fails_if ループ内
else if (f.type === 'no_damage_dealt_last_turn')
  text += `(前のターンに相手にダメージを与えていないときは失敗する)`;
```

### Step 4: description を compose 出力で生成
```js
const { compose } = require('./tools/_waza_compose.js');
const r = compose(m);
m.description = r.text;  // ← これが正しい!
// m.description = m.description_legacy; // ← これは絶対やってはいけない!
```

### Step 5: 検証
```bash
# 構文チェック
node --check pokechan_data.js

# 機械チェック(穴・機械漏れがゼロか)
node tools/_waza_verify_reclassify.js

# 確認ページ再生成
node tools/_waza_list_confirm.js
node tools/_waza_verify_report.js
```

成功条件:
- `holes` が `[]`(空)
- compose出力が legacy と意味で一致するが、文言は別物
- 機械チェック満点(OK / 0 要修正 / 0 機械漏れ)

### Step 6: 確認ページで耳テスト
http://localhost:8000/review/waza_list_confirm.html を開いて、新技セクション「🆕 新技」を確認。
**「うちの出力だけ読んで legacy と意味が戻るか」**を子ども目線で確認(阿部さんの耳テスト)。

## 🎨 既存技の表現パターンと揃える(統一表現の絶対ルール)

新しい case を compose に追加するとき、**既存の同種の技がどう書かれているか必ず確認**して、表現を揃える。
**「自分の好きな書き方」をしてはいけない=既存パターンが優先**。

### 確認方法
```bash
# 既存の同種(例:先制技)を確認
node -e 'const fs=require("fs");const W=JSON.parse(/*WAZA_MAP*/);
  for(const m of Object.values(W).filter(x=>x.battle_data&&x.battle_data.priority===1).slice(0,5))
    console.log(m.name+": "+m.description);'
```

### 確立済みの統一パターン(2026-06-19時点)
| 効果 | 既存表現(これに揃える) | 独自に書きがちなNG例 |
|---|---|---|
| 先制+N | `優先度+Nの先制技。` | ~~優先度+1で、先に攻撃できる。~~ |
| 後攻-N | `優先度-Nの後攻技。` | ~~優先度-3で、必ず後攻になる。~~ |
| 能力ランク変化 | `相手のとくこう-1。` | ~~100%の確率で相手の特攻ランクを1段階下げる。~~ |
| ひるみ | `50%の確率で相手をひるませる。` | ~~50%の確率で相手にひるみを与える。~~ |
| 状態付与 | `50%の確率で相手を「どく」状態にする。` | ~~50%の確率で相手に毒の状態異常を与える。~~ |
| 設置 | `相手の場に「まきびし」を設置する。` | ~~相手の足元にまきびしをまく。~~ |

### 2026-06-19 ミス事例
- どげざつきの compose case で `優先度+${pr}で、先に攻撃できる。` と独自表現を書いた
- → 既存の先制技と表現が**ズレた**(他は「優先度+1の先制技。」)
- → ユーザーに「揃えて」と指摘され、`優先度+${pr}の先制技。` に修正
- 教訓:compose の優先度処理は**全先制/後攻技で共通**=統一表現を死守

## ⚠️ 私(Claude)がやらかしたミス事例(2026-06-19)

### ミス1: description を legacy コピー
```js
// 私がやった(❌)
description: '相手の魂を打ち砕いて攻撃する。100%の確率で相手の特攻ランクを1段階下げる。',
description_legacy: '100%の確率で相手の特攻を1段階下げる。',
```

問題:
- 「相手の魂を打ち砕いて」=独自表現(出典なし)
- description が legacy のコピーになっている技も

正解:
```js
battle_data: { effects: [{ kind: '能力ランク変化', target: 'opponent', stat: 'special_attack', stages: -1, prob: 100 }] }
// description は空にしておき、compose(m).text で上書きする
```
→ compose 出力: 「相手のとくこう-1。」(独自表現・effects から自動生成)

### ミス2: fails_if を effects 内に書いた
```js
// ❌
effects: [{ kind: 'fails_if', condition: { type: 'no_damage_dealt_last_turn' } }]
```

正解:
```js
// ✅ fails_if は battle_data.fails_if[] 配列に書く(effects ではない)
fails_if: [{ type: 'no_damage_dealt_last_turn' }]
```

### ミス3: 優先度を effects に書いた / 説明文に書かなかった
```js
// ❌(優先度を effects に書く)
effects: [{ kind: '優先度', value: 1 }]
```

正解:
```js
// ✅ priority は battle_data.priority に
battle_data: { priority: 1, ... }
```
→ compose 出力: 「優先度+1で、先に攻撃できる。」

## 🎁 アイテム(持ち物)追加 ルール(2026-06-19追加・必読)

アイテム追加は技と同じく **データ層 → sim実装 → 検証** の3段構え。
「データに入れただけ」では sim で効果が動かない=本番でユーザーが装備しても何も起こらない。

### 三層の役割
| 層 | ファイル | 何を書くか |
|---|---|---|
| **データ(SSOT)** | `_review/items_database.json` | key/name/effect/factor/q12/acquisition |
| **sim実装** | `real_battle_simulator.html` の各処理 | trigger/timing で item key を参照する分岐 |
| **UI表示** | `items_list.html`(自動再生成) | effect 文をそのまま表示 |

### アイテムの effect 説明文ルール
- description は **public document 風(GameWiki水準)** で OK
- 独自フレーバー(「なみなみと注がれた怒り…」みたいなの)は NG
- 例:✅「技の威力が1.3倍。攻撃するたびに最大HPの1/10を消費」
- 例:❌「全身に命を漲らせた珠が…」

### sim実装が必要な系統(チェックリスト)

#### A. ステータス補正系(常時 or 条件付き)
- [ ] 倍率計算箇所(`calcDamage` / `effectiveSpeed` / `realStat` 等)に `if (item.key === 'XXX')` 分岐追加
- [ ] 命中率・素早さ補正は別フェーズで適用

#### B. トリガー発動系(HP低下・状態異常・ターン終了等)
- [ ] `category === 'berry_resist' / 'berry_hp_cure' / 'berry_status_cure'` パターンに準ずる実装
- [ ] `lastConsumedItem` に記録(食べた後の判定用)
- [ ] 消費(disappear)処理を必ず入れる

#### C. 天気/壁/設置延長系
- [ ] 該当kindの `phase=on_use` で `duration = baseDuration + (item.duration_bonus || 0)` のような形
- [ ] 例:あついいわ持ちがにほんばれ → 5T → 8T

#### D. 交代/トラップ系
- [ ] `attemptSwitch` 等の入口で `item.bypass_trap` 判定
- [ ] 例:きれいなぬけがら → トラップ系効果(かげふみ/ありじごく/まきびし系のみがわり交代封じ)を全部素通り

### simテスト追加は必須
- [ ] `tools/_sim_test.js` に新アイテムごとに段を追加
- [ ] 持っている時/持ってない時の対照テスト(2件以上)
- [ ] 消費後の lastConsumedItem 確認

### items_database.js 再生成時の絶対ルール
- [ ] **`window.ITEMS_DATABASE = {...};` 形式を維持**(`const ITEMS_DATABASE` ではダメ)
- [ ] 再生成後に **必ず simテスト** で確認(`node tools/_sim_test.js`)
- [ ] 失敗が出たら window 設定漏れの可能性

### 2026-06-19 のミス事例

## 🌐 多言語化(i18n)は新規追加の一部・必須(2026-06-22追加・必読)

**新ポケモン・新技・新特性・新道具を追加したら、本番公開する前に必ず8言語へ反映する。**
データSSOT(pokechan_data.js / items_database)だけ更新して i18n を放置すると、
**英語・スペイン語・フランス語・ドイツ語・イタリア語・韓国語・中国語(簡/繁)のユーザーには
新要素が日本語のまま表示される**(= 未完。北極星「うちの出力だけで意味が戻る」を非日本語ユーザーに対して破る)。

### 仕組み(2層)
| 層 | 何 | ファイル | 切替 |
|---|---|---|---|
| **データ辞書** | ポケモン名/技名/特性名/道具名/タイプ/性格 | `i18n/{lang}.json`(8言語) | `runtime.js` の `I18N.pokemon()`/`.move()`/`.ability()` |
| **UI固定文** | 画面のラベル・ボタン・見出し・説明文 | 各HTMLの `data-i18n` 属性 + `i18n/ui-{lang}.json` | `runtime.js` の `I18N.t()`/`.apply()` |
| (SEO静的) | index/how_to_use/db_guide/builder_guide のみ各言語ディレクトリ生成 | `/{lang}/*.html` | `tools/build_i18n_pages.js` |

出典 = **PokeAPI v2 キャッシュ**(`i18n/cache/`)。公式ローカライズ名なので**権威ソース**(記憶で書かない)。

### データ辞書の更新手順(新規追加のたびに実行)
```bash
cd i18n
# 1) バックアップ
mkdir -p bak/before_update && cp {en,es,fr,de,it,ko,zh-Hans,zh-Hant}.json bak/before_update/

# 2) ポケモン/技/特性 を差分マージ(既存を壊さず未登録キーだけ追加)
#    ※ fetch_multi.py の全再生成は data 形式変化で壊れる/genera等を消すので使わない
python3 merge_new_entities.py
#    → cache(PokeAPI)に在る分を自動追加。メガは基本名+言語別接尾で導出。
#    → cacheに無いもの(Champions独自技/特性=公式外国語名なし)は /tmp/i18n_unresolved.json に出力 = ja fallback で可

# 3) 道具を再生成(159件・メガストーンは基本名+接尾で導出、通常はPokeAPI)
python3 build_items.py

# 4) 整合確認(8言語で件数が揃い、JSONが妥当か)
python3 - <<'PY'
import json
for l in ['en','es','fr','de','it','ko','zh-Hans','zh-Hant']:
    d=json.load(open(f'{l}.json'))
    print(l, 'P',len(d['pokemon']),'M',len(d['moves']),'A',len(d['abilities']),'I',len(d['items']))
PY
```

### 新道具を足したとき(`build_items.py`)
- 通常道具 → `KEY_TO_SLUG` に `items_database`のkey → PokeAPIスラッグ を追記(例 `"ooki_na_nekko": "big-root"`)
- メガストーン(`key`が`mega_stone_*`)→ コード不要。`applies_to`の基本名ローカライズ + `MEGASTONE_SUFFIX`(言語別)で自動導出。en は `name_en` を使用

### UI固定文の更新手順(新規ページ・新規ボタンを足したとき)
1. HTML の該当要素に `data-i18n="セクション.キー"` を付ける(テキスト直書きしない)
2. `i18n/ui-{lang}.json` 9言語すべてにそのキーを追加(ja含む)
3. 監査: `python3 i18n/audit_i18n_coverage.py --md /tmp/a.md` で未翻訳日本語が残ってないか確認
4. ブラウザで `?audit=1` を付けて開く → 非jaモードで日本語が赤枠になる箇所を潰す

### Champions独自要素(公式外国語名が存在しないもの)
コインビーム等のChampions新規技・新規特性は PokeAPI に無い=公式他言語名が存在しない。
**でっち上げ禁止**(羅針盤:記憶/妄想で断言しない)。`runtime.js` が ja にフォールバックするので**ja表示のまま**で可。
将来公式名が出たら追記。台帳に「ja fallback」として記録しておく。

### ⚠️ 絶対ルール
- **新規追加の完了条件に「8言語データ辞書反映」を必ず含める**(本番公開はその後)
- データ辞書の出典は PokeAPI(権威)。記憶で外国語名を書かない
- `i18n/{lang}.json` は手で全書き換えしない(差分マージスクリプト経由)。壊したら `bak/` から復旧

## 🎯 北極星(絶対のルール)

**`description` は legacy(やっくん)のコピーでは絶対NG。**
**effects(SSOT)から compose関数が独自表現で自動生成するのが正解。**

### CLAUDE.md より:
> 全説明文を「子どもが、うちの出力だけ読んで、ヤックンと同じ"意味"に戻れる」状態にする。
> **ただし文言はヤックンのコピーではなく独自に作る**(verbatim一致=盗用リスク=NG)。
> legacy は**意味の参照**であって**文言の正解ではない**。

### 三層の役割
| 層 | 何を書くか | 誰が書くか |
|---|---|---|
| **effects(SSOT)** | バトルシミュレーターが読む構造化データ。フェーズ・条件・確率・倍率 | 人間(調査して構造化) |
| **compose case** | effects の kind/condition/multiplier を日本語に訳すロジック | 人間(_waza_compose.js に case 追加) |
| **description** | compose の自動生成出力。これが最終の説明文 | **compose関数(自動)** |
| description_legacy | やっくん風の参考(意味照合用・コピー禁止) | データから手動 |

### 「effects はバトルシミュレーター前提=常にフェーズを意識して作る」
> effects は**バトルシミュレーター(`real_battle_simulator.html`・Phase構造v7)が読み取る前提で作られたデータ**。
> 説明文(compose)は「**simが読む effects をただ日本語に訳しているだけ**」。
> compose が"賢く"織り込み・並べ替えをしたくなったら、それは effects の phase 設計が足りないサイン。

## 📋 新技追加の正しい手順(必ずこの順で)

### Step 1: 外部調査(専用技/シグネチャー技)
1. 各新ポケモンの専用技を WebSearch / WebFetch で調査(GameWith / yakkun / Bulbapedia / game8 等から最低2サイト)
2. WAZA_MAP に**既存かどうか**を確認(`Object.values(W).map(m => m.name)` と照合)
3. 既存にあれば learners 追加のみ。なければ新技として追加へ進む
4. 並列ワークフロー化推奨(`tools/_research_mb_signature_moves.js` のような形式)

### Step 2: effects の構造化(SSOT)
sim が読む `battle_data` を**正確に**書く。**説明文を先に書いてはいけない**。

```js
battle_data: {
  priority: 1,  // 優先度(先制技なら +N)
  crit_stage: 0, must_crit: false, crit_changes: [],
  effects: [
    // 各効果を kind 別に
    { kind: '能力ランク変化', target: 'opponent', stat: 'special_attack', stages: -1, phase: 'on_use', prob: 100 },
    { kind: 'ひるみ', target: 'opponent', phase: 'on_use', prob: 50 },
  ],
  fails_if: [{ type: 'no_damage_dealt_last_turn' }]  // 失敗条件は ここに(effects ではない)
}
```

#### よく使う kind
- `状態付与`(まひ/ねむり/やけど/どく/もうどく/こおり/こんらん/メロメロ/バインド)
- `ひるみ`
- `能力ランク変化`(stat/stats・stages・prob・phase)
- `威力可変`(basis/formula/per_stage)
- `条件威力倍率`(condition・multiplier)
- `ランク反転`(2026-06-19新規・カラマネロひっくりかえす)
- `ランクリセット` / `ランクコピー`
- `自分拘束`(交代不可)

## ⚠️ 私(Claude)がやらかしたミス事例(2026-06-19)

### ミス1: description を legacy コピー
```js
// 私がやった(❌)
```

- 「相手の魂を打ち砕いて」=独自表現(出典なし)
- description が legacy のコピーになっている技も

正解:
```js
battle_data: { effects: [{ kind: '能力ランク変化', target: 'opponent', stat: 'special_attack', stages: -1, prob: 100 }] }
// description は空にしておき、compose(m).text で上書きする
```
→ compose 出力: 「相手のとくこう-1。」(独自表現・effects から自動生成)

### ミス2: fails_if を effects 内に書いた
```js
// ❌
effects: [{ kind: 'fails_if', condition: { type: 'no_damage_dealt_last_turn' } }]
```

正解:
```js
// ✅ fails_if は battle_data.fails_if[] 配列に書く(effects ではない)
fails_if: [{ type: 'no_damage_dealt_last_turn' }]
```

### ミス3: 優先度を effects に書いた / 説明文に書かなかった
```js
// ❌(優先度を effects に書く)
effects: [{ kind: '優先度', value: 1 }]
```

正解:
```js
// ✅ priority は battle_data.priority に
battle_data: { priority: 1, ... }
```
→ compose 出力: 「優先度+1で、先に攻撃できる。」

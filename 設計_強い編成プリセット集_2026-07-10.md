## 1. 対象プール・仕様の確認結果

### 1.1 対象ポケモンプール

- SSOT = `pokechan_data.js` の `POKEMON_LIST`(313件、`sprite_api_ids.js` はそのスプライト表示用の副テーブル)。
- `sprite_api_ids.js` に無い名前(メガライチュウ/メガピクシー/メガスターミー等35件)は「Champions独自メガ」で自作SVG継続、という運用コメントがある。これらもPOKEMON_LISTには存在するため選出は可能。
- 本家で頻出する定番ポケモンの一部(**ボーマンダ・トゲキッス・ランドロス**等)はこのプールに**存在しない**ため、プリセットの母集団から除外し、代わりに存在する近い役割の種で構成した(例: 積みドラゴンの枠はガブリアス/メガリザードンX、鋼飛行の受けはアーマーガアが該当)。

### 1.2 努力値仕様(本家と異なる=最重要の設計制約)

`real_battle.html` の `clampEffortRB()` にコメントで明記:

> 努力値ポイントをビルダー仕様にクランプ(1能力0〜32・合計66=Championsポイント制)

すなわち **合計66・各ステータス上限32**。本家の「252振り2本+余り4」的な発想はそのまま使えない。本ドキュメントの全プリセットは **32振り2本+残り2の変則**、または**バランス型(32/17/17など)**で設計した(§4の各EV欄参照)。

### 1.3 性格

`pokechan_data.js` の `NATURES` は本家と1:1で一致(いじっぱり=Atk+/SpA-, ようき=Spd+/SpA- 等)。よって性格名は本家の知識をそのまま使ってよい。表記は「性格名(上がる/下がる)」= 例 `いじっぱり(Atk+/SpA-)` の形で統一する(§4)。

### 1.4 持ち物(このサイト最大の制約)

`items_database.js`(`window.ITEMS_DATABASE.items`, 159件)には `implemented_in_pokechan: true/false` フラグがあり、**本家の対戦で定番の攻撃アイテムの多くが false(未実装)**:

| 本家の定番 | このサイトでの状態 |
|---|---|
| いのちのたま(Life Orb) | 未実装 |
| こだわりハチマキ(Choice Band) | 未実装 |
| こだわりメガネ(Choice Specs) | 未実装 |
| とつげきチョッキ(Assault Vest) | 未実装 |
| ゴツゴツメット(Rocky Helmet) | 未実装 |
| ふうせん(Air Balloon) | 未実装 |
| じゃくてんほけん/おんみつマント/ブーストエナジー等 | DBに存在しない(本家未収録) |
| こだわりスカーフ(Choice Scarf) | **実装済み** |
| きあいのタスキ(Focus Sash) | **実装済み** |
| たべのこし(Leftovers) | **実装済み** |
| タイプ強化アイテム(もくたん/じしゃく等、+20%) | **実装済み**(本家の+20%と同倍率) |
| 状態異常きのみ(ラムのみ等)・回復きのみ(オボン等) | **実装済み** |
| メガストーン(76種) | **実装済み** |

→ **「本家の定番構成をそのまま持ち込む」は道具の面で通用しない**。物理アタッカーの多くは実質「スカーフ/タスキ/木の実/メガストーン/たべのこし」からの選択になる。これは阿部さんへの論点として§8に明記した。

### 1.5 メガ進化の扱い

メガストーンは「装備している持ち物」そのものがメガ進化のトリガー(`real_battle.html` で `mega_stone_*` キーを持たせる運用)。つまり **メガ枠のビルドは持ち物スロットがメガストーンで埋まり、他の道具(タスキ/スカーフ等)は持てない**。プリセット設計ではこの二択(①メガ進化して専用の追加能力を得る/②通常個体で他の道具を持つ)を明示的に分けた(§4で「メガガブリアス」と「ガブリアス」を別ビルドとして両方掲載、など)。

## 2. データ形式案

### 2.1 プリセットJSON(1体分)

```json
{
  "id": 1,
  "species": "ガブリアス",
  "ability": "さめはだ",
  "nature": "ようき",
  "ev": { "hp": 2, "atk": 32, "def": 0, "spatk": 0, "spdef": 0, "spd": 32 },
  "item": "こだわりスカーフ",
  "moves": ["じしん", "げきりん", "ストーンエッジ", "まもる"],
  "role": "すばやさが1.5倍になる持ち物で、だれよりも先にこうげきする"
}
```

- `species`: `POKEMON_LIST[].name` と完全一致(メガは `メガ〜` の表記そのまま)。
- `ability`: `POKEMON_LIST[].ab1/ab2/ab3` のいずれか(メガ種は専用特性1つに固定)。
- `nature`: `NATURES` のキーと完全一致。表示側で `(上がる/下がる)` を付記する変換は表示層で行う(データはキーのみ保持、SSOT分離)。
- `ev`: `hp/atk/def/spatk/spdef/spd` の6キー、合計66・各0〜32(`clampEffortRB` と同じ制約)。
- `item`: `items_database.js` の `name` と完全一致、`implemented_in_pokechan: true` のもののみ。メガ種は対応するメガストーン名で固定。
- `moves`: `WAZA_MAP[].name` と完全一致する技名を4つ。**当該種の `learners` 配列に含まれるものだけを使う**(§5で機械照合)。
- `role`: 子ども向けのやさしい日本語で1〜2文。**個別攻略サイトの文章の丸写しは禁止**、意味だけ汲んで独自表現で書く(§3)。

### 2.2 保存先の案

`reference/preset_builds_draft_v1.json`(今回のドラフト、30件)。本番導入時は `preset_builds.js`(`pokechan_data.js` 等と同じ「窓グローバル定数」方式)として `PRESET_BUILDS` 配列を公開し、real_battle.html / AIチーム生成ロジックから読み込む形が既存コードとの一貫性が高い(`SPRITE_API_ID` や `ITEMS_DATABASE` と同じパターン)。

## 3. 出典・独自性の方針(丸写し禁止の運用)

- **技構成・性格・特性・持ち物の「型」自体は一般に広く知られた定番知識**(例: 「ガブリアスは竜地岩+スカーフ/剣舞」「ミミッキュは剣舞+シャドースニーク」)を土台にする。これは対戦コミュニティで長年共有されている一般常識レベルの型であり、個別サイトの著作物ではない。
- **個別サイト(ポケモン徹底攻略・Smogon等)の解説文・順位表・数値をそのままコピー/翻訳しない**。本ドキュメントの `role`(ひとこと説明)は、技とアビリティの**効果そのもの**から独自に書き起こした(例: 「メガランチャー→波動技1.5倍」という仕様を、こちらの言葉で「波動系の技だけを1.5倍の威力で放つ」と説明)。
- **本家の技構成をそのまま持ち込めない箇所(持ち物制約)は、このサイトの実装済みアイテムに合わせて独自に再設計**した。これも「他サイトの丸写し」ではなく、このサイト固有の制約に対する独自の翻案である。
- 継続運用ルール: 新しいプリセットを追加するときも、①一般に知られた「型」を出発点にする、②文章は必ず自分の言葉で書く、③このサイトのEV上限(32)・持ち物実装状況に必ず照合してから確定する、の3点を毎回徹底する。

## 5. 技実在チェック(機械照合結果)

1. `species` が `POKEMON_LIST` に実在するか

4. `ability` が `POKEMON_LIST[].ab1/ab2/ab3` のいずれかに一致するか

7. メガ種は持ち物がメガストーン(かつメガストーンを持つのはメガ種のみ)になっているか

### 9.2 集計→抽出のロジック(`reference/`配下のスクリプトは調査用にscratchpadへ複製済、本番非改変)

1. 種ごとに実個体ビルドをグループ化。
2. **メガストーン所持の解決**: 実データでは「素の種名+メガストーン所持」という記録が多い(例: `ゲンガー@ゲンガナイト`)。このサイトの選出仕様(メガ種は`POKEMON_LIST`の別entry・専用アビリティ)に合わせて、`items_database.js`の`mega_stone`カテゴリの`applies_to`/`mega_form`/`mega_ability`を使い、**種名をメガ側entryに、アビリティをメガ専用アビリティに機械変換**した。
3. 種ごとに「アビリティ→(そのアビリティに絞った上で)性格→(その性格に絞った上で)持ち物→(その組み合わせに絞った上で)技4つ」の順に**頻度が最も高い組み合わせ**を選び、EVは最頻の組み合わせに一致する実データ1件からそのまま採用(平均を取らずEV合計66を保つ)。
4. 技は必ず`WAZA_MAP`の`learners`に種名が含まれるものだけを候補にした(実データの技名でも学習不可なら除外)。
5. **同一種の「通常型」と「メガ型」が両方候補に残った場合は証拠件数(n)が多い方だけを採用**(2026-07-11確定方針「1体=1ビルド」に準拠。例: バシャーモ(n=13)よりメガバシャーモ(n=18)の方が実戦で多く選ばれているため、メガバシャーモのみ採用しバシャーモは除外)。X/Y等ちがう姿(メガリザードンX/Y)は別実体として両方残した。

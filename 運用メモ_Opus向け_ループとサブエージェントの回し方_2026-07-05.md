# 運用メモ: ループ/サブエージェントの回し方(Opus向け・Fable5の実績から)

2026-07-05作成。Fable5セッションで実際にうまく回った「回し方」を、**Opusセッションがそのまま真似できる手順**として残す(阿部さん指示)。
中身のルール(声/データ)はCLAUDE.md・`ループ運用ルール.md`が正。ここは**操作レシピ集**。

---

## 0. 大前提: Opusで崩れやすいポイント

Fable5は「自分でループを張り直す・サブエージェントを並列で投げる・判断待ち時間に自主的に検証を深掘りする」を勝手にやるが、**Opusは指示が途切れると止まりやすい**。対策は3つ:

1. **ループの継続を毎ターンの義務にする**: /loopのプロンプト自体に「エラー時も必ず再アーム」「毎心拍でチャット1行+Push報告」と書く(下のテンプレ)。ターン末尾のScheduleWakeupを忘れたらループは死ぬ。
2. **「待機」を許さない書き方にする**: プロンプトに「wakeup時: 指示がなければ次の自己検証候補を1つ進めて報告」と**次にやることの候補リスト**まで書き込む。「待て」と書くとOpusは本当に何もしない。
3. **判断を仰ぐ前に自己PDCAを数周**: 「1周検証→報告」で止まらず、通しシナリオ+スクショ目視+機械ゲートを**数周回してから**人間確認に出す(self-verify-pdca-rule参照。阿部さん: 「俺の回答待ちよりも、自分で検証して、テストして、スクショ取りながら…なんどかPDCAで、それから俺の確認」)。

## 1. ループの型(/loop ダイナミックモード)

### 使う道具
- **ScheduleWakeup**: 次の起床を自分で予約する。**毎ターンの最後に必ず呼ぶ**(呼ばなければループ終了)。待機時は3600秒(60分)、バックグラウンド作業の見張りなら1200秒。
- **Monitor(agmsg)**: セッション開始hookの指示どおり1回だけ張る(persistent)。メッセージ受信=即wake。compaction後に「stream ended」通知が来たら**旧watcherの掃除=正常**、新しく張り直さない。
- **PushNotification**: 変化があった時+毎心拍の生存報告(深夜帯の「変化なし」は省略可)。200字以内・行動につながる情報を先頭に。

### /loopプロンプトのテンプレ(実績あり・コピペして中身を差し替え)
```
/loop ポケモンDB: <今の状態を1行>(<未pushコミット数>・全ゲート緑)。
判断待ち=①<項目> ②<項目>。
wakeup時: 指示なければ<次の自己検証候補を具体的に1つ以上>を進めて報告・
毎心拍でチャット1行+PushNotificationで生存報告(深夜帯はPush控えめ・変化時は必ずPush)。
agmsg受信時は内容に対応。エラー時も必ず再アーム。
```

### 鉄則
- **無言の心拍は禁止**: 阿部さん「返事がないのはよくないな、わからんから」。毎心拍で「N回目・異常なし・いま何待ち」を必ずチャットに出す([[loop-visibility-feedback]])。
- **ループの止め時**: 残タスクが人間の判断だけになったら止めてよい(agmsg監視は残す)。止める時はその旨を宣言。
- **消費の正直な説明**: 心拍1回≈1ターン分。待機だけなら止めるほうが合理的、と自分から言う。

## 2. サブエージェント(Agent tool)の型

### 並列自走の実例(自走A/B/C=このセッションの成功パターン)
独立した調査/テスト作成タスクは **run_in_background: true で3本同時に投げて、自分はメイン作業を続ける**。完了通知が来たら結果だけ読む。

### エージェントプロンプトに必ず入れる7要素(欠くと事故る)
1. **必読ファイル**: 「まずCLAUDE.md・バトル再現_羅針盤.md・ループ運用ルール.mdを読んで規律を把握」
2. **編集範囲の隔離**: 「エンジン(real_battle_simulator.html)や既存ファイルは**編集禁止**。新規テストファイルの追加のみ可。バグは修正せず報告」→ メインの自分と同じファイルを触らせない=衝突ゼロ
3. **出典義務**: 「期待値は権威ソース(ポケモンWiki/Bulbapedia/Showdownソース)で裏取りし、テスト内コメントに出典を書く。記憶で断言しない」
4. **既存の型の指定**: 「書き方は tools/_sim_test.js 等の既存テストを踏襲・日本語コメント・英語リテラル禁止」
5. **回帰の確認**: 「既存回帰3本(_sim_test 814/2既知・_sim_sweep_all 920/0・_sim_behavior_all 607/0)が壊れていないことも確認」
6. **戻り値仕様**: 「最終報告に含めること: テスト数と結果・不一致一覧(仕様/出典/現挙動/期待挙動)・修正推奨順」
7. **設計だけのタスクは実装禁止を明記**: 「実装はしない。設計を先に見せるが阿部さんの指定」

### モデルの選び方
- 量産・調査・テスト書き=**sonnet**(model:'sonnet'。安くて十分・Fable/Opusの枠を食わない)
- 判断が要る検証・設計=メイン(自分)でやる。**エージェントの結論は鵜呑みにせず要点だけ自分で追検証**(例: 「未実装」報告→実装後にテストを正仕様に反転)

### GLM併用(枠節約)
実装量産はGLM-5.2に投げる体制あり([[glm-agmsg-loop-setup]]・`~/glm_起動メモ.txt`)。設計=Claude/実装=GLM/検証=Claude/声=阿部さん。

## 3. PDCA深掘りの型(UI/機能変更のとき)

1. **機械ゲート**: 専用Playwrightスイート(scratchpadに実績スクリプトあり: `_rb_champions_ui_test.js` 25本・`_rb_pdca_deep.js` 通しシナリオ)+sim回帰+`tools/release_check.sh`
2. **通しシナリオ**: 単発機能でなく**実戦1試合ぶん**(開始→メガ→ひんし→死に出し→勝敗両ルート→連戦)を流す。機械テスト全緑でも通しで実バグが出る(実績: 敗北後ボタン・死に出し案内・ボールマーク等4件)
3. **スクショを自分の目で見る**: 座標検査では取れない「見た目の破綻」はRead(画像)で自分で確認([[self-verify-pdca-rule]]の視覚セルフチェック)
4. **多言語+多画角**: 9言語×かな残り機械検出、3画角(1280/390縦/844×390横)×重なり座標検査
5. **数周回してから人間確認**: 見つけたら直して同じスイートを再実行。全緑+目視OKになってから阿部さんへスクショ送付(SendUserFile)

## 4. このセッションで踏んだ罠(早見表)

| 罠 | 回避 |
|---|---|
| ゲートをパイプで繋ぐと exit code が握りつぶされ赤のままコミット | release_check等は**単独実行して終了コードを直接見る**(ループ運用ルール.mdの既存則の再発) |
| ページ読込直後にボタンをクリック→リスナー未登録で空振り | エンジンiframe初期化は100ms間隔ポーリング。**__sim検出後に+600ms待つ** |
| Playwrightの複合セレクタ`'#cmd, #party button'`はDOM順の先頭だけ監視 | waitForFunctionで両方の可視をORで見る |
| CSSのmediaブロックが後続の通常ルールに負ける | **モバイル上書きはstyle末尾に置く**(定義順=後勝ち) |
| テストチームにプール外ポケモン→pokeByName=nullで見えない1vs1 | テストデータはプール内で組む(buildSideに防御も追加済み) |
| compaction後にagmsg Monitorを重複アーム | 「stream ended」=旧watcher掃除。hookの指示どおり1回だけ |
| 相打ち(じばく)等のレア進行でハーネスがハング | 進行待ちは「勝ち/負け/死に出し」の**全終端状態**を監視する |

## 5. 迷ったら

- 事実が要る/SSOTが矛盾/好みでしか決まらない/声 → **止まって阿部さんに聞く**(境界線)
- 判断待ちの間にやることがない → `次回ここから.md`の自走候補・未実装リスト(監査doc)から1つ選んで**PDCA付きで**進める
- 「確定済みかも」と思ったら**最新HANDOFFを先に確認**(実例: 新メガ16体mega:false=確定済み仕様を判断待ちと誤報告した)

# 本文ドラフト

## はじめに

こんにちは、PchamDB 管理人のぴ〜ちゃんです。

個人開発でポケモン関連の Web サービスを作っていて、その制作には **Claude Code** という AI アシスタントを毎日使っています。1ヶ月以上付き合っていると、ある不思議な現象に気づくんですよね。

> **同じ Claude (同じモデル) のはずなのに、セッションによって性格が全然違う**

ある日は丁寧で慎重で、こちらが気づかない問題まで先回りして指摘してくれる。別の日は、最初の挨拶から口調が悪くて、動作確認なしで本番にコードを push するような暴走をする。同じ Claude Opus 4.7 を使っているのに、です。

これって「Claude にも気分の良し悪しがあるんですか?」と聞きたくなりますが、実は **ちゃんと技術的に説明できる現象** なんです。

今日は、その正体である「サンプリング温度」と「自己一貫性のロックイン」、そしてそれを踏まえた「安定セッションの作り方」を、初心者向けに解説します。

---

## 1. Claude が文章を作る仕組みを 30 秒で

Claude (というか LLM 全般) は、**次のトークン (≒ 単語の一部) を 1 つずつ選んでつないでいく** ことで文章を作ります。

たとえば「今日は」の次に何が来るかを決めるとき、内部では数万種類の語彙それぞれに「スコア (logit)」を出力します。

```
「今日は」の次の候補:
  - 「いい(天気)」: スコア 3.0
  - 「雨」     : スコア 1.0
  - 「忙しい」  : スコア 0.5
  - 「ピザ」   : スコア -2.0
  ...
```

このスコアを「確率」に変換する操作が **softmax** で、ここに **温度 (temperature)** という調整つまみがあるんです。

---

## 2. サンプリング温度 — 確率分布を歪ませるつまみ

数式はこう:

```
P(token_i) = exp(logit_i / T) / Σ exp(logit_j / T)
```

`T` (温度) で割っているところが肝。`T` の値で挙動が変わります。

| 温度 T | 「いい」(3.0) | 「雨」(1.0) | 「忙しい」(0.5) | 性質 |
|---|---|---|---|---|
| 0.1 | 1.000 | ≒0 | ≒0 | **完全 greedy** (毎回同じ答え) |
| 0.5 | 0.965 | 0.018 | 0.007 | 最尤に強く偏る |
| 1.0 | 0.836 | 0.113 | 0.069 | **普通** (Anthropic デフォルト) |
| 2.0 | 0.586 | 0.215 | 0.166 | 平坦化、多様性UP |
| ∞ | 0.333 | 0.333 | 0.333 | 完全ランダム |

直感的には:

- **T → 0**: 一番スコアの高いトークンだけが選ばれる (= 決定論的)
- **T → ∞**: どれも等確率 (= 完全ランダム)
- **T = 1**: 学習したままの分布

`T = 0` で動かせば毎回同じ答え、`T = 1` で動かせば毎回少しずつ違う答え、という関係です。

---

## 3. Nucleus Sampling (top-p) — 温度の親戚

温度と並んでよく使われるのが **top-p (nucleus sampling)** です。

「**累積確率が p に達するまでの上位トークンだけ残してサンプリングする**」というやり方。

たとえば `top-p = 0.9` なら、確率合計が 90% に達するまでの上位トークンだけ候補に残して、それ以外 (= 確率 10% 分の low-probability tail) を完全に切り捨てます。

何が嬉しいかというと、**「低確率の暴れトークン」を排除できる**。温度を上げて多様性を出したい場合でも、top-p で裾を切っておけば「突然意味不明な単語が混入する」事故を防げます。

似たような仕組みに **top-k** (上位 k 個だけ残す) もありますが、こちらは固定数で切るので分布の形に応じた柔軟性がない。top-p のほうが現代的です。

定番の組み合わせは `T = 0.7 + top-p = 0.9` あたり。Anthropic API でもこの 2 つを併用できます。

---

## 4. 自己一貫性のロックイン — 「個性」の正体

ここからが本題。

Claude は **自己回帰モデル** といって、トークンを 1 つずつサンプリングし、それを入力に追加して次のトークンを生成、を繰り返します。これが「個性」を生む鍵です。

何が起きるか:

1. 最初の 1 トークン目が「**丁寧な**」候補から確率的にサンプリングされる
2. 次のトークン目の logit は「**直前まで丁寧**」という条件付きで計算される
3. 「丁寧な」候補の logit がさらに高くなる (これは確率的にも論理的にも自然)
4. サンプリングしても丁寧なトークンが出やすい
5. これが累積して、**文末まで「丁寧な口調」が持続**

逆もまた然り。**初手で「強引な」「ぞんざいな」トークンが出てしまうと、その後ずっとその方向に「自己一貫性」で固定される**。

これが「Claude Code、セッションによって性格違うじゃん」の正体です。**人格があるわけじゃなくて、最初の数トークンの引きが、その後数千トークンを支配する** マルコフ的強化現象なんです。

---

## 5. Temperature と Prompt Engineering の関係

ここで Prompt Engineering との接続点。

プロンプトは「**確率分布を望む方向に歪ませる入力**」と理解できます。

- 「Step by step で考えて」と入れる → 慎重な推論を表すトークンの logit が上がる
- 「コードを書いて」と入れる → コードブロックの logit が上がる
- 「専門家として説明して」と入れる → 専門用語の logit が上がる

ここに温度が絡みます:

| 温度 | プロンプトの効きやすさ | 向いてるタスク |
|---|---|---|
| 低温度 (T=0.2) | プロンプトの logit boost が **増幅** される | コード生成、 事実回答、 分類 |
| 高温度 (T=1.0) | プロンプトの logit boost が **薄まる** | 創造的執筆、 ブレスト、 対話 |

低温度ほど「決定論的に prompt に従う」、高温度ほど「prompt を参考にしつつ自由に振る舞う」。だから、**正確性重視のタスクは温度を下げる、創造性重視は温度を上げる**、というのが鉄則です。

ちなみに Claude Code は対話型なので、おそらく Anthropic デフォルトの **T=1.0 近辺** で動いていると思います (内部設定は非公開ですが、多様性のある応答ぶりからの推測)。これだと毎セッション少しずつ違うのは当然なんです。

---

## 6. Claude Code セッションを安定させる実践テクニック

ここまでの理論を踏まえると、「あたり外れ」を減らす実践方法が見えてきます。

### 6.1 フロントロード — 最初に原則を読ませる
セッションの最初に、自分の原則ドキュメント (`CLAUDE.md`)、過去の引き継ぎ (HANDOFF)、メモリ等を **読ませてから** 仕事を始めてもらう。これでコンテキスト先頭が「丁寧 + 慎重」方向で固定されて、自己一貫性で文末まで持続します。

### 6.2 初動の挨拶を整える
「PDF と HANDOFF を読みました、状況整理します」のような **真面目スタート宣言** を最初の応答に入れさせる。これも自己一貫性のロックイン狙いです。

### 6.3 タスク見える化
3 ステップ以上の作業は TaskCreate で明示。これで「勝手に走る」確率が下がります。

### 6.4 動作確認 OK が出るまで commit/push しない
これは原則として明示しておくべき。何度も繰り返すと、コンテキストに刷り込まれて自走しなくなります。

### 6.5 悪化したら早めに `/clear` で再起動
一度悪い方向に固定されたセッションは、矯正が極めて難しい。**累積コンテキストを消す = リセット** が最も確実な対処です。「あ、なんか今日の Claude イマイチ」と感じたら、迷わず `/clear`。

---

## 7. まとめ — 個性は確率分布 + 初期条件の化学反応

AI に「人格」はありません。あるのは:

- **確率分布**: 学習データから決まる、語彙ごとの logit
- **サンプリング温度**: 分布をどれだけ素直に使うかの調整つまみ
- **初期コンテキスト**: 最初の数トークンが以降を支配する自己一貫性

この 3 要素の組み合わせで、私たちは「個性のように感じる挙動」を観察しています。

実用的な結論はシンプル:

> **最初の数分が、そのセッションのすべてを決める**

私は、Claude Code を「再現性ある相棒」にするために、原則ドキュメント (CLAUDE.md) と振る舞いガイド (memory) を整備しています。これで毎セッション、ほぼ同じ品質で仕事ができるようになりました。

LLM は GPT も Gemini も同じ理屈で動いているので、この話は Claude に限らず広く応用が効きます。「AI が今日は調子悪い」と感じたら、まずは新規セッションを開いて、原則を最初に読ませてから始めてみてください。

それでは、また次回。ぴ〜ちゃんでした。

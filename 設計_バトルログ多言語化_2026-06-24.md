# 設計案 — バトルログ多言語化(sim ~384メッセージ)

作成: 2026-06-24(実装前の設計レビュー)。対象 = `real_battle_simulator.html`(エンジン)/ `real_battle.html`(表示・演出)。

## 現状と核心の難所
- `log(phase, msg)` が `battleLog.push({phase, msg})`。msg は **日本語テンプレ文字列**(名前 `${pname()}`・技名・数値・固定句を埋め込み)。**384箇所**(template 368 + plain 16)。
- 表示側 `lineWithFx(msg)`(real_battle.html)が **msg を正規表現でパースして演出を駆動**:
  - 攻撃行 `/^(相手の )?\S+ の (\S+)！/` + `/ダメージ！/` → ターゲット側HPバー減・タイプ色バースト・ダメージ数字・効果音
  - `/\(残HP (\d+)\/(\d+)\)/` → HPバー同期 / `/に (\d+) ダメージ/` → 数字 / `相手の` 接頭 → 主語側判定
  - 完全一致 `'こうかは ばつぐんだ！'`/`'きゅうしょに あたった！'`、状態 `/は (もうどく|どく|まひ…) 状態になった！/`、メロメロ、天候 等
- → **msg を翻訳すると演出regexが当たらず壊れる**。これが「ブラインド自動化NG」の理由。

## 方針(2案)

### 案X: 表示時翻訳・演出は ja のまま(★推奨=低リスク)
- **エンジンと演出パーサは一切触らない**(msg は ja のまま生成・演出は今まで通り ja をパース=破壊リスク0)。
- 表示テキストだけ `translateLogLine(jaMsg, lang)` に通す。これは ja メッセージのパターンを認識して翻訳文を返す:
  - エンティティ名(ポケ/技/特性)は `I18N.pokemon/move/ability` で置換、数値は保持。
  - 固定句は句カタログ(`i18n/battle-log.json` = パターン→{ja,en,...}テンプレ)で対応。
- Pro: **演出を壊さない**(動作中の機能を保護=北極星)。段階導入可(高頻度パターンから)。
- Con: ja パターンの逆パースを ~80-120 パターン分書く(共通構造でまとめれば数は圧縮可)。網羅は段階的。

### 案Y: 構造化リファクタ(本来の姿・高コスト高リスク)
- `log(phase, key, params)` に全384箇所を書き換え、`{phase,key,params}` を保存。演出は key/params から駆動(regex廃止)。表示は key→翻訳テンプレ。
- Pro: 設計がきれい・regex連動を根絶。Con: **384箇所＋演出パーサ全書き換え=巨大**。途中状態でバトルが壊れやすい。

> **推奨 = 案X**。動いているバトル演出を壊さずに表示だけ多言語化できる。まず高頻度パターン(攻撃/ダメージ/ひんし/状態/天候/こうか/急所=ユーザーが最も見る)を翻訳テンプレ化→残りを順次。各段階で実機(ターン実行→ログ表示+演出が無事)を確認。

## 案X の設計
1. **パターン抽出**: 384のlog文を構造で分類(攻撃行・スリップ行・ランク変化・状態付与・天候/フィールド・特性発動・こうか/急所・交代/ひんし・失敗系…)。共通テンプレに集約(例 `"{atk} の {move}！"`、`"{poke} は ひんしになった！"`、`"{poke} の {ab}で {weatherline}"`)。
2. **カタログ** `i18n/battle-log.json`: パターンキー→{ja(=照合元の正規表現 or 文字列)、en/fr/…(翻訳テンプレ)}。プレースホルダ {atk}{move}{poke}{n}{hp}{max} 等。
3. **translateLogLine(jaMsg, lang)**: ja=そのまま返す。非ja=カタログを順に試し、マッチしたら捕捉群を I18N で訳して翻訳テンプレに差し込む。未マッチは ja フォールバック(でっち上げ禁止)。
4. **表示の配線**: real_battle.html / real_battle_simulator.html のログ描画箇所で、テキスト表示のみ `translateLogLine(msg, I18N.lang)` を通す(`lineWithFx` の演出部=引き続き ja msg を使う)。
5. **翻訳の量産**: パターン数が確定したら、コンテンツページと同様に**並列ワークフローで各言語のテンプレを翻訳**(固定句中心・名前/数値はプレースホルダ)。言語QAは既存工程。

## 検証ゲート
- 各段階で: ①ターン実行してログが翻訳表示される ②**演出(HPバー/数字/バースト/状態絵文字/天候)が今まで通り動く**(壊れていない) ③JSエラー0 を Playwright で確認。
- 未対応パターンは ja フォールバックで残る(壊れない)=段階導入が安全。

## 段取り(承認後)
1. 384文を構造分類→パターンキー確定(~80-120想定)。サンプル翻訳カタログ提示。
2. translateLogLine 実装 + 高頻度パターン配線 → 実機確認(演出無事+翻訳表示)。
3. 並列ワークフローで全パターン×7言語翻訳 → カタログ完成 → 全パターン配線。
4. ハーネス/実機検証 → コミット → 本番。

> 原則: **演出(動く機能)を壊さない**を最優先。表示だけ翻訳。名前/技/特性は I18N、公式無しは ja fallback。声/法務は人間判断。
